14세기, 한 수도사가 복잡한 신학 논쟁에 칼을 들이댔습니다. "불필요한 가정을 **면도날로 잘라내라**!" 700년이 지난 지금도 과학자, 의사, 개발자들이 이 원칙을 따릅니다. 단순함이 진리에 가까운 이유는 뭘까요?

<!-- step:story -->

1320년대, 영국 오컴 마을의 프란체스코 수도사 **윌리엄**은 고민에 빠졌습니다. 당시 신학자들은 신의 존재를 증명하려고 복잡한 논리를 쌓아 올렸죠. "천사는 몇 종류인가?", "영혼은 몇 개의 층으로 나뉘는가?" 같은 질문에 수십 가지 가정을 추가했습니다.

윌리엄은 이렇게 물었습니다. "정말 이 모든 가정이 필요한가?" 그는 **필요 없는 가정을 제거**하는 원칙을 제시했습니다.

> "같은 현상을 설명하는 두 이론이 있다면, **단순한 쪽을 선택하라**."

이 원칙은 당시 교회와 충돌했지만, 과학혁명의 씨앗이 됐습니다. 갈릴레오는 복잡한 천동설 대신 단순한 지동설을 선택했고, 뉴턴은 천체의 운동을 **단 3가지 법칙**으로 설명했죠.

현대 의학도 이 원칙을 따릅니다. 여러 증상이 있을 때, 하나의 병으로 설명되면 그걸 먼저 의심합니다. 불필요한 진단을 면도날로 잘라내는 거죠.

<!-- step:reveal -->

**오컴의 면도날 핵심 원리:**

**"Entities should not be multiplied beyond necessity."**
(필요 이상으로 가정을 늘리지 마라)

**실생활 적용 예시:**

1. **의료 진단**
   - 여러 증상 → 하나의 병으로 설명 가능하면 그걸 먼저 의심
   - 불필요한 복잡한 진단 제거

2. **버그 수정**
   - 복잡한 버그 원인보다 **오타, 설정 실수**를 먼저 확인
   - "가장 단순한 실수"가 가장 흔한 원인

3. **음모론 판단**
   - 간단한 실수로 설명 가능한데 거대한 음모를 주장한다면?
   - 단순한 설명부터 검토

4. **코드 설계**
- 10개 클래스로 해결할 걸 50개로 만들지 마라
   - **KISS 원칙**(Keep It Simple, Stupid)과 연결

**주의할 점:**
- 단순하다고 **항상 옳은 건 아님**
- 때로는 복잡한 설명이 진실일 수도 있음
- 출발점일 뿐, 절대 법칙은 아님

**오늘부터 실천**: 문제를 마주했을 때, "가장 단순한 설명은 뭘까?"를 먼저 물어보세요. 버그를 만나면 복잡한 시스템 오류보다 **오타와 설정**을 먼저 확인하세요. 오컴의 면도날은 진리를 찾는 방법이 아니라, **사고를 시작하는 좋은 출발점**입니다!

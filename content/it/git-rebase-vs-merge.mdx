<!-- step:cinematic-hook -->
코드 리뷰 중.
히스토리가 지하철 노선도처럼 꼬여 있었다.
"누가 이렇게 했어?"

<!-- step:scene -->
브랜치를 합칠 때마다
"Merge branch 'feature/login'" 커밋이 생겼다.
3개월 뒤 히스토리는 추적 불가능.

<!-- step:dialogue:senior -->
"merge 대신 rebase 써봤어?"

<!-- step:dialogue:junior -->
"히스토리가 깔끔해진다던데...
근데 위험하다고 들었어요."

<!-- step:narration -->
시니어가 화이트보드에 그림을 그렸다.
merge는 모든 흔적 보존.
rebase는 일직선 정리.

<!-- step:dialogue:senior -->
"일기장을 정리하는 두 가지 방법.
낙서 흔적까지 남기거나,
깔끔하게 다시 쓰거나."

<!-- step:impact -->
**Merge** → 안전하지만 복잡
**Rebase** → 깔끔하지만 위험

<!-- step:narration -->
규칙은 간단했다.
**혼자 작업하는 로컬 브랜치** → rebase OK
**이미 push한 공개 브랜치** → rebase 절대 금지

<!-- step:dialogue:senior -->
"이미 push한 브랜치를 rebase하면
팀원들 작업이 망가져.
협업 충돌 폭발."

<!-- step:dialogue:junior -->
"로컬에서만 쓰면 안전하겠네요."

<!-- step:reveal-title -->
🔀 Git rebase vs merge 차이
로컬 브랜치는 rebase로 정리,
공개 브랜치는 merge로 안전하게.

<!-- step:outro -->
**협업 충돌 90% 감소.**
push 전에는 rebase, push 후에는 merge.

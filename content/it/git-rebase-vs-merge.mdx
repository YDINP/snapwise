팀 프로젝트에서 브랜치를 합칠 때, 커밋 히스토리가 복잡하게 얽혀서 누가 뭘 했는지 추적이 불가능했던 경험 있나요? 아니면 반대로 너무 깔끔하게 정리했다가 **다른 팀원의 작업이 사라진** 적은요? `merge`와 `rebase`를 언제 쓰냐에 따라 결과가 완전히 달라집니다.

<!-- step:story -->

두 팀이 같은 프로젝트를 진행했습니다.

**A팀**은 매번 `git merge`를 사용했습니다. 브랜치를 합칠 때마다 병합 커밋이 자동 생성됐죠. 3개월 뒤 히스토리는 마치 **지하철 노선도**처럼 복잡했습니다. 하지만 누가 언제 뭘 했는지는 명확히 추적할 수 있었습니다.

**B팀**은 `git rebase`만 고집했습니다. 커밋 히스토리를 일직선으로 정리하니 깔끔했죠. 하지만 어느 날 팀원이 실수로 이미 push한 브랜치를 rebase했습니다. 다른 팀원들이 pull하자 충돌이 폭발했고, 일부 작업이 사라졌습니다.

**C팀**은 두 가지를 상황에 맞게 사용했습니다. 로컬에서 정리할 땐 `rebase`, 공개 브랜치를 합칠 땐 `merge`. 히스토리는 적당히 깔끔하면서도 안전했습니다.

3개월 뒤, C팀만 배포 전 rollback을 단 5분 만에 해냈습니다. 히스토리가 명확했으니까요.

<!-- step:reveal -->

**Merge vs Rebase 완벽 비교:**

**Merge의 특징**
- 두 브랜치의 히스토리를 **그대로 보존**
- 병합 커밋(merge commit)이 자동 생성
- 협업 시 **안전하고 추적이 쉬움**
- 히스토리가 복잡해질 수 있음

**Rebase의 특징**
- 커밋 히스토리를 **일직선으로 정리**
- 깔끔한 로그를 원할 때 유용
- **이미 push한 브랜치에는 절대 사용 금지!** (팀원의 작업이 사라질 수 있음)

**언제 뭘 쓸까?**
- 공개 브랜치(main, develop) 합칠 때 → **`merge`**
- 로컬에서 정리할 때 → **`rebase`**
- 협업 중인 feature 브랜치 → **`merge`**

<!-- step:action -->

**오늘부터 실천**: `rebase`는 **혼자 작업하는 브랜치**에서만 쓰세요. 이미 push한 커밋을 rebase하면 팀원들의 히스토리가 꼬입니다. 공개 브랜치는 무조건 `merge`로 합치는 습관을 들이면 **협업 충돌을 90% 줄일 수 있습니다**!

뷔페와 맞춤요리 중 뭘 선택하시겠어요? 뷔페는 정해진 메뉴를 다 가져다주지만 먹기 싫은 것도 섞여 있고, 맞춤요리는 원하는 것만 받지만 주문이 복잡합니다. **REST와 GraphQL**이 바로 이 차이입니다.

<!-- step:compare -->

두 개발팀이 같은 모바일 앱을 만들었습니다.

**A팀 (REST API)**

`/users`, `/posts`, `/comments` 같은 엔드포인트를 만들었습니다. 하지만 메인 화면 하나를 그리려면 **3개 API를 연속 호출**해야 했죠.

```
GET /users/123        → { id, name, email, address, phone, ... }
GET /posts?user=123   → [ { id, title, content, ... }, ... ]
GET /comments?post=1  → [ { id, text, ... }, ... ]
```

모바일 네트워크에서 딜레이가 쌓이자 사용자들이 "느리다"며 이탈했습니다.

**B팀 (GraphQL)**

클라이언트가 **필요한 데이터만 골라서** 한 번에 요청했습니다.

```graphql
query {
  user(id: 123) {
    name
    posts {
      title
      comments { text }
    }
  }
}
```

네트워크 요청은 1/3로 줄었고, 앱이 빨라졌습니다. 하지만 문제가 생겼습니다. REST처럼 **HTTP 캐시를 쓸 수 없어서** 캐싱을 직접 구현해야 했죠.

**C팀 (하이브리드)**

간단한 CRUD는 REST, 복잡한 데이터 조합은 GraphQL. 각자의 장점을 살렸습니다.

<!-- step:example -->

**실제 사용 예시:**

**REST가 적합한 경우:**

```
GET /products         # 상품 목록 (캐싱 중요)
POST /orders          # 주문 생성 (간단한 CRUD)
DELETE /cart/items/5  # 장바구니 삭제
```

**GraphQL이 적합한 경우:**

```graphql
# 대시보드: 사용자 정보 + 최근 주문 + 알림 한 번에
query Dashboard {
  user { name, avatar }
  recentOrders(limit: 5) { id, total, status }
  notifications { count, unread }
}
```

<!-- step:reveal -->

**REST vs GraphQL 핵심 비교:**

| 항목 | REST | GraphQL |
|------|------|---------|
| **요청 방식** | 여러 엔드포인트 (`/users`, `/posts`) | 단일 엔드포인트 (`/graphql`) |
| **데이터 반환** | 고정된 구조 (뷔페) | 클라이언트가 선택 (맞춤요리) |
| **오버페칭** | 자주 발생 (불필요한 데이터도 전송) | 거의 없음 |
| **타입 시스템** | 없음 (별도 정의 필요) | 스키마 기반 내장 |
| **캐싱** | HTTP 캐시 자동 활용 | 직접 구현 필요 (Apollo Client 등) |
| **학습 곡선** | 낮음 | 중간 |
| **네트워크** | 여러 번 요청 | 한 번에 해결 |

**REST를 선택하세요:**
- 캐싱이 중요한 경우 (상품 목록, 공지사항)
- 간단한 CRUD API (회원가입, 로그인)
- 팀이 GraphQL 경험이 없고 빠른 출시 필요

**GraphQL을 선택하세요:**
- 모바일 앱처럼 네트워크 효율이 중요
- 복잡한 데이터 관계 (SNS 피드, 대시보드)
- 프론트엔드가 유연한 쿼리를 원할 때

<!-- step:action -->

**오늘부터 실천**: 새 프로젝트를 시작하기 전, **"데이터 구조가 얼마나 복잡한가?"**와 **"네트워크 효율이 중요한가?"**를 먼저 물어보세요. GraphQL은 유연성을, REST는 단순성을 제공합니다. 두 가지를 섞어 쓰는 것도 정답입니다!

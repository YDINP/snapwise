API를 설계할 때 가장 먼저 만나는 갈림길. "REST로 갈까, GraphQL로 갈까?" 잘못 선택하면 나중에 **전체 시스템을 뒤집어야** 할 수도 있습니다. 두 길의 차이를 미리 알고 출발하세요.

<!-- step:story -->

두 개발팀이 같은 모바일 앱을 만들었습니다.

**A팀은 REST API**를 선택했습니다. `/users`, `/posts`, `/comments` 같은 엔드포인트를 만들었죠. 하지만 메인 화면 하나를 그리려면 **3개 API를 연속 호출**해야 했습니다. 모바일 네트워크에서 딜레이가 쌓이자 사용자들이 "느리다"며 이탈했습니다.

**B팀은 GraphQL**을 선택했습니다. 클라이언트가 필요한 데이터만 골라서 한 번에 요청했죠. 네트워크 요청은 1/3로 줄었고, 앱이 빨라졌습니다. 하지만 문제가 생겼습니다. REST처럼 HTTP 캐시를 쓸 수 없어서 **캐싱을 직접 구현**해야 했죠.

**C팀은 하이브리드**를 선택했습니다. 간단한 CRUD는 REST, 복잡한 데이터 조합은 GraphQL. 각자의 장점을 살렸습니다.

1년 뒤, C팀의 앱이 가장 빠르고 유지보수도 쉬웠습니다. "만능 솔루션은 없다"는 걸 알았으니까요.

<!-- step:reveal -->

**REST vs GraphQL 핵심 비교:**

| 항목 | REST | GraphQL |
|------|------|---------|
| **요청 방식** | 여러 엔드포인트 | 단일 엔드포인트 |
| **데이터 반환** | 고정된 구조 | 클라이언트가 선택 |
| **오버페칭** | 자주 발생 (불필요한 데이터도 전송) | 거의 없음 |
| **타입 시스템** | 없음 (별도 정의 필요) | 스키마 기반 내장 |
| **캐싱** | HTTP 캐시 활용 | 직접 구현 필요 |
| **학습 곡선** | 낮음 | 중간 |

**REST를 선택하세요:**
- 캐싱이 중요하거나, 간단한 CRUD API일 때
- 팀이 GraphQL 경험이 없고 빠른 출시가 필요할 때

**GraphQL을 선택하세요:**
- 모바일 앱처럼 네트워크 효율이 중요할 때
- 복잡한 데이터 관계가 있고, 프론트엔드가 유연한 쿼리를 원할 때

<!-- step:action -->

**오늘부터 실천**: 새 프로젝트를 시작하기 전, "데이터 구조가 얼마나 복잡한가?"와 "네트워크 효율이 중요한가?"를 먼저 물어보세요. **GraphQL은 유연성을, REST는 단순성**을 제공합니다. 두 가지를 섞어 쓰는 것도 정답입니다!
